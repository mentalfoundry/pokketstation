# PocketStation hardware notes

## CPU

ARM7TDMI (ARM/THUMB), Atmel-fabricated. Variable clock via `CLK_MODE` (see below and `core/src/clk.c`) - a 16-entry table confirmed against real hardware (`PMFrequency`/`SetCpuSpeed`), from 32.768kHz (mode 0, the default/idle rate) doubling each step up to ~4MHz (mode 7), with mode 8 and above all aliasing the same ~8MHz max rate. (An earlier version of this table copied an earlier, unconfirmed source's CPU-frequency table values instead, which read ~2x higher at every corresponding index - not independently confirmed, see the standing note above.)

## Memory map

| Range | Size | Purpose |
|---|---|---|
| `0x00000000` | 2KB | RAM. `0x000–0x1FF` kernel, `0x200–0x7FF` user. |
| `0x02000000` | virtual | FLASH1 — the currently selected/running app, mapped from FLASH2. |
| `0x04000000` | 16KB | BIOS ROM. Kernel around `0x1E00`, GUI around `0x2200`. |
| `0x06000000` | — | Flash control I/O (`F_CTRL`, `F_BANK_FLG`, `F_WAIT`). |
| `0x08000000` | 128KB | FLASH2 — physical flash, 15 blocks. |
| `0x0A000000` | 0x14 | Interrupt controller: hold(+0x0, R), status(+0x4, R), enable(+0x8, W, ORs in), mask(+0xC, W, ANDs matching bits out of enable), acknowledge(+0x10, W, clears matching hold+status bits). See "Interrupt controller, timers, and RTC" below. |
| `0x0A800000`+ | 0x30 | 3 timers, 0x10 bytes apart: period(+0x0), count(+0x4), control(+0x8, bits0-1 = clock divisor, bit2 = enable). See below. |
| `0x0B000000`+ | 0x8 | `CLK_MODE` - CPU/timer clock speed control (bits 0-3 index a `CPU_FREQ` table, readback ORs in a "steady"/PLL-locked bit `0x10`). A real BIOS dump polls this (`LDR/TST #0x10/BEQ`) before touching flash control, waiting for the clock to stabilize after a speed change. Modeled in `core/src/clk.c` - `psemu_run` scales its effective cycle rate by the currently-programmed mode (see the resolved HELLO/audio-speed issue below); readback always reports "stable" since speed changes are instantaneous in this emulator. |
| `0x0B800000` | 0x10 | RTC: mode(+0x0), control/adjust(+0x4), time(+0x8, R), date(+0xC, R). See below. |
| `0x0C800000`+ | — | IR registers (protocol/send-receive mode at `+0`, beam on/off at `+4`). Exact bit-level IR timing is **unverified** —  |
| `0x0D000100` | 128B | LCD VRAM. |
| `0x0D800000` | 0x10 | IOP power control (): IOP_CTRL(+0x0, unmodeled, no known effect), IOP_STOP/IOP_STAT(+0x4, W/R, sets bits), IOP_START(+0x8, W, clears bits), IOP_DATA(+0xC, unused by the real BIOS). Bit5 = Sound Enable - audio requires both this AND `DAC_CTRL` bit0. See `core/src/iop.c`. |

BIOS ROM and FLASH2 are only readable in 16/32-bit units; RAM is freely 8/16/32-bit.

## LCD

32×32 pixels, 1 bit per pixel. VRAM is 128 bytes (32 rows × 32-bit words). Bit 0 of each word is the leftmost pixel; 0 = white, 1 = black. Hardware refreshes at roughly 32Hz once the CPU writes a row.

## Buttons

5 face buttons (Up, Right, Down, Left, Fire) plus a physical reset, read as bits 0–4 of `INT_INPUT` at `0x0A000004`.

## App file format: PSX Title Sector

PocketStation apps are **not** VMS (that's Dreamcast VMU) or GME (that's the DexDrive whole-card dump format) — don't reuse that terminology in code or docs. The real container is the **PSX Title Sector**, an 80-byte header followed by icon data, an optional snapshot, a function table, and the executable body:

- Offset `0x00–0x4F`: standard PS1 memory-card title/icon header fields.
- Offset `0x50–0x5F`: PocketStation-specific — icon frame counts, an `"MCX0"`/`"MCX1"` identifier at `0x52`, entry point + THUMB-mode flag (bit 0) at `0x5C`.
- If the identifier is `MCX1`: an `0x800`-byte snapshot follows (saved ARM register/RAM state), then the function table and executable body. `MCX0` skips the snapshot.

Source: the documented file-header/icons page.

## Confirmed against a real BIOS dump

`tools/inspect.c` (a small headless CLI, not part of the automated test suite) loads a real BIOS + a real app extracted from a memory card and free-runs the interpreter, logging mode transitions and stopping on any unrecognized opcode. Real dumps are never committed - see `testdata/` (gitignored). Findings so far, running two different real PocketStation apps for 2M+ instructions each with no unimplemented-opcode hits:

- Reset genuinely starts in Supervisor mode, ARM state, and the very first ~12 instructions visit FIQ then IRQ then back to SVC - textbook ARM startup code initializing each mode's banked SP in turn. Confirms the banked-register model works for real code, not just hand-written tests.
- The SWI vector is at `0x00000008` and does live in RAM (as expected - the BIOS populates the low-memory vector table itself; there's no separate hidden vector ROM).
- After boot, the kernel switches to User mode and runs a periodic SVC↔USR↔`SWI` cycle - **but the USR-mode code runs from a fixed BIOS address (`0x04001AF8`), not from FLASH1 (`0x02000000+`) where a loaded app lives.** Confirmed by loading two different real apps and a "no app" run and getting byte-for-byte identical instruction traces - the kernel hadn't yet reached whatever logic reads the flash header and jumps to the app's own entry point, at least within a 5M-instruction window. Don't read early mode-switching/SWI activity as "the app is running" without checking the PC is actually inside FLASH1.
- The app-independent cycle does write to LCD VRAM (`psemu_framebuffer_dirty()` goes true during the run, always to an all-zero/blank pattern) - real, working end-to-end LCD I/O, but this is the kernel's own idle/menu behavior, not evidence the app has drawn anything.
- Real register-offset addressing (`LDR/STR reg,[reg,reg]`) is what caught a genuine dispatch bug: the single-data-transfer class check incorrectly required the I-bit (bit 25, a data field selecting immediate-vs-register offset) to be 0, when the real class boundary is only the top 2 bits (`27:26 == "01"`). No hand-written test had exercised register-offset addressing through `arm_execute`'s dispatch before this.
- **PocketStation apps are frequently bundled as multi-block linked memory-card files together with the game's regular PS1 save**, not as a standalone single 8KB block. A naive per-block scan for the `MCX0`/`MCX1` magic will find the right *first* block but silently miss the rest of the app if you don't follow the card's actual directory (block 0): each of the 15 data-block frames is 128 bytes with allocation state at offset `0x00` (`0x51`=first/solo, `0x52`=middle, `0x53`=last, `0xA0`=free), total filesize at `0x04`, and a next-block link at `0x08` (a **little-endian u16, 0-based among the 15 data blocks - add 1 to get the physical block number**, `0xFFFF` sentinel = end of chain). Extracting only the first block still parses fine (the header's in there) and still boots/runs without crashing, but any code/data the app references past its first 8KB reads back as zeroed flash - worth checking chain length before assuming a boot problem is a CPU bug.

### App-selection and dispatch, reverse-engineered from a real BIOS disassembly (Capstone)

The kernel's app-selection routine (BIOS file offset `~0x1A00-0x1B00`) does the following, in order - this is read directly off real disassembly, not inferred from docs:

1. Calls a helper (`0x1BC8`) that returns a **selected app-slot index** from RAM: `*(u16*)0x000000D0` if nonzero, else `*(u8*)0x000000CE`. If both are zero, no app is selected and the kernel takes an early-exit fallback (this is the "idle" path observed before any of this was understood).
2. **`FLASH2` (`0x08000000`) carries the exact same directory structure as a PS1 memory card** - not just the app's own block, the whole card layout is mirrored into flash. Given the selected slot index `N`, the kernel reads the directory frame at `FLASH2 + N*128` and walks the chain forward via the frame's next-block link (same `+1` convention as the memory card), accumulating a bitmask of every physical block belonging to that app (`mask |= 1 << block_index` per link hop).
3. Writes that bitmask to `FLASH_CTRL + 8` (`F_BANK_FLG`, `0x06000008`) and the value `2` to `FLASH_CTRL + 0` (`0x06000000`) - programming a **hardware bank-select window**: FLASH1 (`0x02000000+`) is not a fixed alias of "block 0", it's a virtual 16-slot window onto FLASH2. **Resolved**: `F_BANK_FLG` says which *physical* 8KB blocks are enabled; a second, previously entirely-unimplemented register, `F_BANK_VAL` (`0x06000100`-`0x0600013C`, 16 words, reset value 0), says, **per physical block**, which virtual bank slot (0-15) it appears at - `table[p]=v`, deliberately the "backwards" direction from a typical page table. **Implemented** in `core/src/flash.c`/`flash.h`: `flash_resolve_physical_bank` reverse-searches `bank_val[]` for a physical block claiming a given virtual slot, live on every FLASH1 access (no more eager caching on the commit write). When `F_BANK_VAL` is untouched (its reset value, `0` for every physical bank) - which is every case this emulator's own app-selection routine has ever been observed to trigger, since it only ever writes `F_BANK_FLG`, never `F_BANK_VAL` - resolution falls back to this emulator's previously-validated behavior: the enabled physical blocks treated as one contiguous run starting at the lowest-numbered enabled block. This closes the previously-open question of whether the window is a simple linear offset or compacts non-contiguous blocks in chain order: it's neither in general - it's a real, explicit per-block remapping table, and the "linear offset" behavior already implemented was only ever a special case (an all-zero `F_BANK_VAL`) that happens to cover every chain observed so far because they're physically contiguous. See `test_flash_bank_val_remapping` for regression coverage of a genuine non-contiguous/reordered mapping, and `test_flash_bank_select` for confirmation the fallback is byte-for-byte unchanged.
4. Re-reads the slot index, then reads the entry point directly from `FLASH2 + slot*8192 + 0x5C` (the same entry-point field already documented above, just read from its real physical position this time instead of a relocated offset-0 copy).
5. Some further validation (calls into two small Thumb helpers reached via classic ARM↔Thumb `add lr,pc,#1 / bx lr` interworking trampolines - decode these as Thumb starting at `trampoline_addr+8`, not as ARM, or the rest of the disassembly is garbage) checks the block's own header a second time (`FLASH2 + slot*8192 + 2`, a 5-bit field there minus 15, times 128 - looks like it locates the function table based on icon-frame count) before deciding whether to jump fresh or resume from an `MCX1` snapshot.
6. Clears user RAM (`0x200-0x7FF`), switches `CPSR` to User mode, sets User SP to `0x800` (top of RAM), and does `BX` to the entry point read in step 4.

### The real SWI (syscall) handler, disassembled

The vector table at RAM `0x00000000-0x0000001C` is 7 identical `LDR PC,[PC,#0x18]` entries (the standard ARM "vector → literal pool → real handler" idiom) plus one `MOV R0,R0` filler at `0x14`; the actual handler addresses live in the literal pool right after, at `0x20-0x3C` (reset, undef, SWI, prefetch-abort, data-abort, reserved, IRQ, FIQ, in that order). For this BIOS dump, SWI's literal (`0x28`) resolves to `0x04001528`. That function:

1. Saves `r1-r12,lr`, reads `SPSR` into a scratch register, and tests its `T` bit to compute the *original* SWI instruction's own address in `lr` (`-2` if the caller was in Thumb, `-4` if ARM - both correctly undo the `pc+2`/`pc+4` return-address convention to point back at the `SWI` itself).
2. Re-reads that instruction's low byte as a **syscall number**, looks up a function pointer from a dispatch table whose base address is stored at RAM `0x000000E0` (`table[syscall_number]`), and calls it via an interworking `BX` (so a Thumb-tagged target correctly switches the CPU into Thumb mode for the call).
3. Returns via **`LDM SP!,{r1-r12,lr,pc}^`** - not `MOVS/SUBS PC,LR`. The `^` suffix on an `LDM` that includes `PC` is architecturally identical to the data-processing `Rd=15,S=1` idiom (restore the whole `CPSR` - mode, `I`/`F`/`T`, flags - from the current mode's `SPSR`, not just load `PC`) but is a completely separate encoding path. **This was a real, confirmed emulator bug**: `exec_block_transfer` decoded but silently ignored the block-transfer S-bit entirely. Fixed - see the CPU test `test_arm_ldm_exception_return` for the regression coverage, and the "App-selection" section above for how this was found (a real Thumb `SWI` issued by the app itself was returning in the wrong mode, landing on data that only decoded sensibly as Thumb while the CPU was stuck in ARM mode).

**Status**: with the app-slot selector, flash bank-select, and this `LDM^` fix all in place, a real BIOS + a real app now runs cleanly for 20M+ instructions (including simulated button presses) with zero unimplemented-opcode hits, executes multiple genuine syscalls through the real dispatch mechanism above, and correctly threads ARM/Thumb mode across all of them. The LCD stays blank throughout this window - not yet known whether that's because the app hasn't reached its "draw something" logic yet, needs a trigger not yet simulated (real button timing, IR, something else), or there's a further bug still to find. That's the next open question, not a settled failure.

### A real, confirmed CPU-architecture bug: Thumb `BL` must tag `LR`'s bit 0

Tracing the halt trap above to its root cause: the app's own first syscall returns correctly via the fixed `LDM^`, continues into what looks like ordinary app code, then hits `BX LR` after a plain Thumb `BL` call - and the CPU incorrectly switches to ARM mode, because `LR` held a plain even return address. **This is wrong even on pure ARMv4T** (verified directly against real ARMv4T architecture behavior, which computes the return address as `(R15 + 2) | 1`) - Thumb `BL` has always tagged the return address's bit 0, precisely so a later `BX LR` correctly stays in Thumb. This is not an ARMv5 `BLX`-only feature, contrary to what I'd assumed earlier in this investigation. Fixed in `exec_long_branch_link` - see `test_thumb_bl_bx_lr_stays_thumb` for the regression case, modeled directly on the real failure.

Fixing this (plus two related memory-map bugs it exposed - `INT_INPUT`/`INT_LATCH` were only handled as single exact-address byte matches, silently losing bit 9 on a 32-bit read; and a second undocumented "ready" register at `0x0B800000`, distinct from the existing `0x0B000000` one) got the same real app running 20M+ instructions with no halt trap at all, versus hitting it within ~5000 instructions before.

### Interrupt controller, timers, and RTC

Implemented in `core/src/intc.c`, `core/src/timer.c`, `core/src/rtc.c`.
   Verified register-by-register against real hardware behavior (using
   official register names like `RTCControl`/`RTCInc`/`RTCClock` alongside
   the informal `RTC_MODE` names). That verification surfaced real,
   confirmed discrepancies between this codebase's original implementation
   and real hardware - the point of doing it at all, since that implementation hadn't been independently validated against real hardware and was informally known to be incomplete in places.

**Interrupt controller** (`0x0A000000`): `hold`, `status`, `enable`, `mask` registers; `intc_irq_asserted`/`intc_fiq_asserted` compute `hold & enable & INT_IRQ_MASK`/`INT_FIQ_MASK` on demand. Register layout, byte-write accumulation, and the four operations (hold/status read-only, enable ORs in, mask ANDs out of enable + latches, acknowledge clears both) match the real register semantics exactly (`INT_LATCH`/`INT_INPUT`/`INT_MASK_SET`/`INT_MASK_CLR`/`INT_ACK` map onto the same four registers).

**A second real, confirmed bug, found by disassembling the real BIOS directly** (going past what any prior documentation covered): an earlier, uncorrected version of this codebase's interrupt-routing function routed `INT_STATUS_MASK` sources (buttons, RTC) into `status` *only*, never `hold` - matching the assumption that `hold` drives real interrupts while `status` is a separate polling-only view. Tracing why the app-dispatch routine (below) was never reached led to disassembling the real IRQ vector handler at `0x04001414` and the periodic system-tick callback it invokes (installed at `0x040036EC` via `SetCallbacks`, RAM `0xFC`) - both explicitly test `hold & enable & INT_RTC` and `hold & INT_BTN_ACTION` respectively, and both branches land on real, working handlers (an RTC acknowledge + day-rollover bookkeeping routine at `0x04000672`, confirmed by tracing it; a button/wake handler at `0x04003784`) that could never execute under that earlier status-only routing, since these bits would never appear in `hold`. Real hardware evidently latches these sources into *both* registers. Fixed `intc_set_line` to do the same; confirmed via instrumentation that both handlers now actually execute in response to simulated RTC ticks and button presses (previously: zero executions, ever). See `test_intc_status_sources_also_latch_hold`.

**Timers** (`0x0A800000`+, 3 channels): control register bits 0-1 select a clock divisor - `0` or `3` = `/2`, `1` = `/32`, `2` = `/512` (the officially documented divider table matches exactly). **A real, confirmed bug**: an earlier version of `timer_tick` in this codebase decremented `count` by raw elapsed cycles directly, ignoring the divisor entirely - any timer configured for `/32` or `/512` would have fired 16x or 256x too often relative to real hardware. Fixed by giving each timer its own cycle accumulator and only decrementing `count` once `divisor` cycles have accumulated (see `test_timer_clock_divisor`). Measured effect on a real BIOS+app trace: mode/IRQ transitions per instruction dropped by roughly 25x after the fix, and LCD-framebuffer writes per instruction went up by over 200x - strong evidence the CPU was previously being starved by a runaway IRQ rate that doesn't reflect real hardware timing. Deliberately *not* modeled: having timer-count reads themselves decrement the count as a side effect of *reading* it - count reads are documented as plain "current value (decrementing)" with no read-side-effect language, so that would be an unconfirmed behavior, and copying it would be adding an unconfirmed bug, not fixing one.

**RTC** (`0x0B800000`): `mode` bit 0 (official `PRGSEL`) is Run (0, ticks at 1Hz, auto-advances the clock every tick) vs Program/pause (1, ticks at ~4096Hz, does *not* auto-advance - used only to let a manual adjust-write step one field at a time without the clock moving underneath it); bits 1-3 (`CNTSEL`) select which BCD field a `control`/`RTC_ADJUST` write adjusts. Two more confirmed, fixed discrepancies:

- **Missing auto-advance.** an earlier, unconfirmed source's own `rtc_tick` callback auto-increments `time` (seconds → minutes → hours → day-of-week, *not* cascading into `date` on a day rollover - a gap in an earlier, unconfirmed source itself that no independent source explains, so it's inherited rather than invented) whenever `mode & 1 == 0`. An earlier version of this codebase's `rtc_tick` only toggled the interrupt line and never advanced the actual time value - real BIOS code reading `RTC_TIME` would see a clock that never moves on its own. Fixed by implementing this exact cascade arithmetic as `rtc_advance_second`, gated the same way.
- **Flat tick rate.** an earlier, unconfirmed source's RTC timer always fires at 1Hz, even in Program/pause mode - the `~4096Hz` behavior documented for paused mode isn't implemented in an earlier, unconfirmed source at all (grep confirms only one `attotime::from_hz(1)` in its entire RTC path). Fixed by giving this codebase two tick-cycle constants, `RTC_TICK_CYCLES_RUN` and `RTC_TICK_CYCLES_PAUSED` (exactly 4096x faster), selected by `mode & 1` each tick.
- **Reset defaults.** An earlier version of this codebase's `rtc_init` hardcoded `time = 0x01000000, date = 0x19990101` (1999-01-01) - this matches the BIOS's well-known "resets itself to Jan 1999" *software* quirk (see below), not the actual silicon power-on-reset value. the real POR values are as `RTCClock = 0x04000000` (day-of-week BCD 4, 00:00:00) and `RTCCalendar = 0x00980101` (1998-01-01) - also confirming that `RTC_DATE` bits 24-31 are an unused/unknown field, *not* a "year-hi"/century byte as an earlier version of this codebase assumed (the real century lives in battery-backed kernel RAM, surfaced only via the `GetBcdDate` SWI). Fixed `rtc_init` to use the officially documented POR values.

The BIOS's own "resets itself to Jan 1 1999" behavior (known as "The RTC Problem" - apparently Sony's software-side workaround for inaccurate clock hardware) is a *separate*, software-driven action taken via the normal `RTC_ADJUST` mechanism, not something to bake into `rtc_init`.

None of these fixes (including the hold-routing bug above) resolved the separate, still-open issue of the app-dispatch routine (`0x04001900-0x04001AFC`, see below) never being reached - confirmed by direct instrumentation before and after each one. They're independently-confirmed correctness fixes regardless.

### RESOLVED: the app-dispatch routine, and a real app, now run

**A real PocketStation app now boots, runs, makes genuine syscalls, survives real timer interrupts, and draws its own deliberate graphics (a bordered play area with a diamond-shaped icon/sprite) - confirmed over a 150M-instruction run with zero unimplemented opcodes and 5.5M instructions of real execution inside the app's own FLASH1 code.** This took two separate fixes on top of everything below, both busy-wait bugs in `core/src/flash.c`'s `flash_ctrl_read8` that a real BIOS/app routine polls expecting a specific bit to read back set:

1. **`FLASH_CTRL+0` (command/commit trigger) is write-command/read-status on real hardware, not a plain mirror.** A real routine writes `2` here, then busy-waits on this *same* address's bit 0 reading back `1` ("ready"). Echoing back the raw command value (bit 0 of `2` is `0`) left that loop spinning forever - this was the dispatch function's own bank-select-commit step (`0x04002E08-0x04002E18` in the directory-scan handler), so nothing past it, including the dispatch function itself, could ever run. Fixed: reads of `+0` now OR in bit 0 (our bank-commit is synchronous, so it's always "ready" immediately).
2. **`FLASH_CTRL+0x10` (the documented `F_WAIT2`, "waitstates, and FLASH-Write-Control-and-Status") wasn't modeled at all** - `FLASH_CTRL_SPAN` stopped at `0xC`. A real app's own flash-write routine polls bit 2 here, expecting it to read back set once the write completes; a default/unmapped read of `0` left *that* loop spinning too, immediately after fixing bug 1 above (FLASH1 execution went from 8259 instructions to 5.5M once this was also fixed). Fixed: extended the span and made `+0x10` always report "not busy" (writes complete instantly here).

Both were found only because real app execution finally got far enough to *reach* them - static disassembly alone would never have surfaced either, since they only matter once a real dispatch/flash-write sequence actually runs. See `test_flash_ctrl_busy_wait_bits` for regression coverage.

**The real unlock, though, was hardware ground truth from the user's own physical PocketStation**, tested directly to answer exactly this question. The confirmed real power-on sequence: 1) "HELLO" renders one letter at a time, 2) a heart icon shows briefly, 3) a beep, 4) a time-setting screen - press Down once, then Action, to continue, 5) a clock screen - Right from there moves to the first app in the list. Also critical: real button taps are only ~40ms - this session's earlier simulated presses held each button for a mere 500 instructions (implied duration far shorter than even a 40ms tap at the real ~4MHz clock, roughly 160000 cycles), which likely failed to register as a real press at all against any debounce/minimum-duration check. `tools/inspect.c`'s `button_sim=3` implements the confirmed sequence with generously-long (150000-instruction) holds and a long enough phase (2.5M instructions) for each animation stage to finish before the next press lands.

Direct instrumentation (tracking every instruction executed in `0x04001900-0x04001AFC`, the app-selection/dispatch function) had shown it was **never entered at all** under any button sequence or slot-index forcing tried before this - PC instead spent the whole run oscillating within `~0x04003xxx`, the BIOS's own menu/event loop (a tight wait on a RAM flag, `0x0400356C`, then dispatch on an action code read from RAM). That period's hypotheses, ruled out along the way, are kept below since each is a real, confirmed fact about the system worth not re-discovering:

Static disassembly hit a hard limit worth recording: **this BIOS freely interleaves ARM and Thumb code**, and the same bytes decode to nonsense in the wrong mode - a constant-propagation script written to auto-find button-mask tests (scanning statically as pure Thumb) silently misdecoded a real ARM function as garbage Thumb and found nothing. Static scans here are only trustworthy when the mode at each address is independently known (e.g. from a real `BX` target's bit 0). This pushed the investigation towards *runtime* instrumentation instead, which tracks mode correctly by construction: a `psemu_debug_current_pc` hook (`core/src/cpu.h`/`cpu.c`) plus an `intc_read8`/`intc_write8` trace gate (`psemu_intc_trace_enabled` in `intc.c`, enabled via `tools/inspect.c`'s `intctrace` flag) log every real INTC access with its real PC.

Hypotheses tested and ruled out so far:

- **Docking (`INT_IOP`, bit 11).** This is a real wake/launch trigger ("Wakeup usually done by IRQ-0 (Fire) and IRQ-11 (Docking)"), and it was never simulated at all before this investigation. Added a `dock` flag to `tools/inspect.c`; confirmed the real docking IRQ handler (`0x04001cf4`) does execute, but dispatch still isn't reached afterward - docking is likely a separate PS1-connectivity feature, not the standalone app-launch trigger.
- **A `button_sim` bug invalidated an early navigation test.** `button_sim` was declared as `argc >= 5 && atoi(...) != 0` - a boolean collapse that silently turned a "2" (Down-then-Fire navigation) argument into the same value as "1" (Fire-only). Fixed to store the real integer; re-ran the navigation-sequence test properly afterward.
- **The interrupt-controller hold-routing bug** (previous section) let the button-press and RTC hold-bit handlers execute for the first time (146-197 button-handler entries per multi-million-instruction run, versus zero before), but didn't unblock dispatch on its own. The button handler in question (`0x04003784`) turned out to just clear a wake/timeout flag, not feed the menu's action-dispatch value.
- **`[r7+0x20]`, initially suspected as a second dispatch source, is actually one slot of a generic icon-blink-animation array** (write sites `0x04002B84`/`0x04002CC0` index into it via a separate counter and happen to land there) - a dead end now corrected; it was never menu-navigation logic at all.
- **A tempting "HOLD is a sticky latch, only ACK clears it" refinement, disproved empirically.** the real register naming (`INT_INPUT` "Raw Interrupt Signal Levels" vs `INT_LATCH` "Interrupt Request Flags") and the real RTC handler's explicit self-acknowledge both suggested `hold` should only clear via an acknowledge write, never on the source de-asserting. Tried it - but the real button-action callback (`0x04003784`) never acknowledges its bit, so a single button press left it latched forever, and the CPU re-entered the IRQ handler on nearly every subsequent instruction (559034 re-entries across one 20M-instruction run - not how a working device behaves). Reverted: both `hold` and `status` clear together on de-assertion, for every source. A real lesson, not just a dead end: a plausible-sounding, partially-evidenced hardware model can still be wrong, and needs a runtime sanity check (does the resulting execution pattern actually look like a working device?), not just internal consistency with the sources that suggested it.
- **INTC trace confirms buttons are never polled via STATUS at all** - across a full navigation-sequence run with every real INTC access logged, the STATUS register's low byte (button bits) never once reads nonzero anywhere in the whole BIOS, even during simulated press windows and even after the hold-routing fix made the bits correctly appear in `hold`. Whatever processes button input, it isn't direct STATUS polling (that mechanism seems reserved for the RTC's wait-for-pulse idiom) - it must go through the HOLD/IRQ-callback path already traced, which is why the search now focuses there.
- **The app/save classification from earlier in this session was re-derived from scratch this round and got it backwards.** Re-checking the real `MCX0`/`MCX1` signature at Title Sector offset `0x52` (the actual, authoritative test, not directory-frame size/naming heuristics) confirms: card frame 1 (7 blocks) and frame 8 (4 blocks) both carry a real `MCX0` signature - genuine PocketStation apps. Frame 12 (1 block) has all-zero bytes at that offset - a plain PS1 save with no PocketStation code at all. Once `F_BANK_VAL` was implemented (see "App-selection and dispatch" above), a temporary trace on writes to `0x06000100`-`0x0600013C` confirmed the real BIOS's app-selection routine *does* write it, at `pc=0x04001A44` (the same routine already documented there), assigning physical block 1 -> virtual slot 0, block 2 -> virtual slot 1, ..., block 7 -> virtual slot 6 - a sequential compact assignment, exactly matching what the existing linear-offset fallback already assumed for this case. With translation now happening for real instead of falling through to a stale assumption, the destination sector correctly resolves to an enabled physical block and the bit-0 check passes.

Confirmed via the full `button_sim=3`/150M-instruction regression (raw `flash_app1.bin`): the retry-loop call count dropped from 1,850,793 to **114** calls, FLASH1 (app code) execution jumped from 18,965,006 to **141,543,936** instructions, and the dispatch/FLASH1-first-entry instruction numbers (`#5,662,784`/`#5,664,188`) stayed exactly unchanged from the pre-`F_BANK_VAL` baseline - confirming this was a strict improvement with zero regression to everything already working. Zero unimplemented opcodes throughout.

### The desktop frontend now runs a real app interactively

Added `psemu_load_flash_image` (`core/include/psemu/psemu.h`) - a public API for loading a raw FLASH2 image (e.g. a whole memory card dump) directly, bypassing `psemu_load_app`'s single-Title-Sector validation. `tools/inspect.c`'s `raw` mode now goes through this too, instead of reaching into the internal `psemu_t` struct. `frontends/desktop/main.c` auto-detects a `PSEMU_FLASH_SIZE`-byte (128KB) input file as a card image; anything else is loaded as a single validated app. Confirmed the desktop build actually works end-to-end: the existing `build/` directory had never been configured with the vcpkg toolchain, so `pokketstation_desktop` had never actually been built despite SDL2 being installed - reconfigured a fresh `build_desktop/` with `-DCMAKE_TOOLCHAIN_FILE=.../vcpkg.cmake`.

Two bugs found immediately from actually trying it interactively:

- **LCD pixels rendered red instead of black.** `SDL_PIXELFORMAT_RGBA8888` packs a 32-bit value as `(R<<24)|(G<<16)|(B<<8)|A` regardless of host endianness. The renderer wrote `0xFF000000` for "on" pixels, intending opaque black, but that's R=0xFF,G=0,B=0,A=0 - pure red with alpha 0. `SDL_RenderCopy`'s default blend mode ignores the source alpha and blits RGB as-is, so it displayed solid red. Fixed to `0x000000FF` (R=G=B=0, A=0xFF).
- **No audio at all** - there was no DAC/sound implementation whatsoever. Added one (`core/src/dac.c`/`dac.h`): two registers at `0x0D800010` - `ctrl` (+0x0, bit 0 = enable) and `data` (+0x4, bits 6-15 = `DACV`, a signed 10-bit two's-complement output level). Real hardware has no square-wave/noise generator or sound DMA channel at all - software produces every tone by writing new `DACV` levels directly to `DAC_DATA` at audio rates (typically via Timer1/IRQ-8), i.e. it's a raw CPU-bit-banged 1-sample-at-a-time DAC with no fixed rate of its own. `dac_tick` resamples the currently-held level (zero-order hold) into a ring buffer at a fixed internal rate, `PSEMU_AUDIO_SAMPLE_RATE_HZ` (8000Hz). New public API: `psemu_get_audio_samples`, drained once per frame in the desktop frontend into an `SDL_QueueAudio` call.

  Wiring up real-time-paced audio initially surfaced what looked like a **third bug**: the desktop frontend's main loop called `psemu_run(ps, 33000)` per ~31ms (`SDL_Delay(31)`), implying only ~1.06MHz of emulated throughput, which seemed inconsistent with the ~4MHz approximation already used by `rtc.h`'s `RTC_TICK_CYCLES`. Changed to `psemu_run(ps, 125000)` (125000/4000000 = 1/32s) to match - **this was wrong**. Real-hardware testing immediately showed on-screen animations (the time-setting screen's blink) visibly too fast after that change. `RTC_TICK_CYCLES` was already documented as "not calibrated against a real reference," so matching a second guess to an uncalibrated first guess wasn't a real fix - it was two unvalidated numbers agreeing with each other. Reverted to `psemu_run(ps, 33000)` (the value already empirically confirmed to look right), and recalibrated the DAC's sample-rate conversion to match it instead (`dac.h`'s `PSEMU_ASSUMED_CPU_HZ = 33000 * 32`, giving a clean 132 cycles/sample) - keep both in sync if either ever changes, since audio pitch/tempo and on-screen timing need to agree with each other.

See `test_dac_basic` for DAC regression coverage.

### RESOLVED: Action button "does nothing while held, confirms on release" now matches real hardware

Reported directly from interactive testing: holding Action mostly didn't register, and specifically - checked directly against a real device - holding Action on the time-setting screen does nothing at all (the blinking year keeps blinking normally); only *releasing* it actually sets the date. Our emulator did the opposite: holding Action stopped the blink entirely.

Ruled out one hypothesis first: a suspected IRQ-storm/livelock from a continuously-asserted HOLD bit was tested directly (`tools/inspect.c`'s `button_sim=4`: hold Fire from instr #0 for the entire run, no release) and **disproved** - PC advances normally, no stuck-PC warnings. Confirmed separately that `ENABLE` never includes bit 0 (Action) across several real-BIOS traces, so Action's `hold` bit was never even contributing to real interrupt delivery in the first place - no storm was possible.

Found the real cause with a targeted test (`button_sim=5`: the confirmed real power-on sequence through the time-setting screen, then hold Action continuously with no release) plus runtime watchpoints on the RAM locations already identified as driving the blink animation (`0x260`, a pulse counter incremented at the tail of the generic system-tick callback; `0x3F0`, the icon-blink-state slot). With Action held, **both froze completely** - confirmed by hashing periodic framebuffer snapshots: identical hash for the entire hold duration, versus alternating between two hashes once fixed. The mechanism: the generic system-tick callback (`0x040036EC`) is a chain of conditional early-exit branches checking `hold` bit-by-bit in a fixed order - IOP, battery, Timer1, **Action**, then (only if none of those matched) STATUS for RTC. With Action's `hold` bit continuously set (a real, physically-held button), the callback *always* takes the Action branch and *never reaches the RTC check* - permanently blocking the RTC-driven redraw path for as long as the button is held. Real hardware evidently doesn't hit this, which reveals what `hold` actually means for buttons: it's a momentary edge pulse marking that a press happened, not a sustained level for the whole physical hold duration (unlike genuine level-triggered hold-only sources like IOP/battery/timers).

Fixed in `psemu_set_buttons` (`core/src/psemu.c`): a press edge still calls `intc_set_line` as before (latching into both `hold` and `status`), but a button still held with no new edge now calls the new `intc_clear_hold_only` (`core/src/intc.c`/`intc.h`) - clearing just `hold`, leaving `status` to keep tracking the live level for any code that polls it directly. Confirmed via the same watchpoints: both `0x260` and `0x3F0` now keep toggling normally throughout a sustained hold. Bonus effect: re-running the full real-app-launch regression test (`button_sim=3` through 150M instructions) now reaches dispatch far sooner (instr #5.65M vs #63.15M before) and the app runs over 3x longer (18.9M vs 5.5M instructions) - the callback no longer gets stuck skipping RTC/redraw processing during simulated button holds elsewhere in the boot sequence either. See `test_button_hold_pulses_not_sustained` for regression coverage.

Also added a minimum button-hold-duration latch to `frontends/desktop/main.c` (`BUTTON_LATCH_FRAMES`, 5 frames) - a real ~40ms tap is only ~1.3 frames at 32Hz, short enough to risk landing entirely between two per-frame keyboard polls and being missed outright; any detected press now reads as held for at least 5 frames regardless of the real tap's exact duration. Kept the temporary button-edge logging in the same file (prints frame number + button state on every press/release) for any further reports.

### RESOLVED: CLK_MODE throughput scaling, and which subsystems should track it

Reported directly from interactive testing: the "HELLO" letter-by-letter animation rendered too slowly, but only while audio (a beep) was actively playing during it - once the audio stopped, rendering sped up to about the expected pace. This went through three iterations before landing on the fix below - worth reading in order since each attempt's failure directly motivated the next.

**Attempt 1.** A write trace (`psemu_clk_trace_enabled`, `memory.c`/`tools/inspect.c`'s `clktrace` mode) on `CLK_MODE` (`0x0B000000`) and `DAC_CTRL` (`0x0D800010`) confirmed the BIOS genuinely sets `CLK_MODE` to mode 7 for the whole HELLO/heart/beep sequence, dropping to a slower mode once `DAC_CTRL`'s enable is cleared. `psemu_run` was changed to scale its effective cycle throughput by `clk_current_hz()`, reasoning that real hardware runs proportionally more cycles per real frame while `CLK_MODE` is elevated - this fixed the animation timing, but feeding the same `CLK_MODE`-scaled raw cycles to `rtc_tick`/`dac_tick` (which assume a fixed reference rate) let them race ahead of real elapsed time during the mode-7 window - reported as the beep "starting too slow and speeding up to sounding ok."

**Attempt 2.** Decoupling RTC/DAC's cycle currency from `CLK_MODE` (converting each step's real elapsed time back into a fixed reference rate) fixed that, but direct measurement (`tools/frame_audio_probe.c`, mimicking the desktop frontend's real per-frame `psemu_run` loop) showed the app's actual audio content-rate *still* scaled with `CLK_MODE` - confirmed directly against real hardware: **the beep played far too fast.** At the time this looked like decoupling Timer specifically (which was still left on raw, CLK_MODE-scaled cycles) wouldn't help, since a quick test of decoupling it too seemed to show no change in the app's DAC_DATA write rate. `psemu_run` was reverted entirely to a plain fixed cycle budget - fixing audio at the cost of reopening the animation bug.

**Attempt 2 was based on a flawed measurement.** Timer1's IRQ is the documented driver for the BIOS's own "GUI" audio-generation loop (`INT_INPUT.8 ;used as Audio square wave IRQ by GUI`) - directly contradicting the "decoupling Timer doesn't help" finding. Redoing that experiment properly, with an exact Timer1-fire counter (not the earlier coarse, aliased sub-slice polling) and by forcibly overriding `CLK_MODE` to different fixed values mid-run, confirmed Timer1's real-time firing rate - and hence the app's actual audio content rate - **does** stay constant when Timer is decoupled from `CLK_MODE` the same way RTC/DAC are, regardless of what `CLK_MODE` is set to.

**Fixed** (`core/src/psemu.c`'s `psemu_run`): the outer loop's overall cycle throughput still scales with `clk_current_hz()` (fixing animation timing), but Timer, RTC, and DAC all now receive a cycle count converted back into the fixed `PSEMU_ASSUMED_CPU_HZ` reference currency via a fractional carry (`ps->real_time_cycle_carry`), rather than raw `CLK_MODE`-scaled cycles - decoupling all three from `CLK_MODE` while the CPU's raw instruction throughput still varies with it. (This specific "decouple Timer too" step was later reverted - see Attempt 4 below - so the test that covered it, `test_clk_mode_keeps_timer_rtc_dac_on_real_time`, no longer exists under that name; `test_clk_mode_scales_run_speed` still covers the throughput-scaling half.)

Also corrected while investigating: `core/src/clk.c`'s `CPU_FREQ` table had been using values roughly double the officially documented table (`PMFrequency`/`SetCpuSpeed`) at every corresponding index - mode 0 = 32.768kHz doubling each step, mode 7 = ~4MHz, mode 8+ = ~8MHz. Updated to match the documentation - real-time pacing (audio, animation timing) should always be checked against the documentation or real hardware, never assumed.

Verified via `tools/inspect.c`'s `button_sim=3` full-launch regression (raw `flash_app1.bin`, 150M instructions, unaffected by any of this since that harness steps the CPU directly rather than through `psemu_run`): dispatch reached at the same instruction count as the pre-CLK_MODE baseline (#5,662,784 vs #5,651,963), FLASH1 app code executes the same total instructions (18,965,006 vs 18,965,272), zero unimplemented opcodes.

**Attempt 4 (superseding Attempt 3's Timer decoupling above).** After Attempt 3 shipped, interactive testing (with a real device to compare against) reported: the beep now played correctly, but tried at 2x/4x its rate (at the user's request, to probe the mechanism) revealed *other* GUI behavior sped up too - specifically the date-setting screen's blink. A new probe (`tools/date_screen_probe.c`, running the real boot sequence through `psemu_run`'s actual per-frame loop and navigating to the date screen via timed button presses) confirmed the visible blink (RAM `0x3F0`) is **not** RTC-driven at all - RTC's own `int_line` toggles far too slowly (~121 frames) to be it - it toggles roughly every 3 frames, tracking the same Timer1 heartbeat that drives audio. This meant Timer1 does double duty as a general GUI tick, not an audio-exclusive IRQ, and decoupling it from `CLK_MODE` (Attempt 3) broke that GUI-tick role even though it fixed audio.

The math confirmed this precisely: with Timer decoupled (pinned to the `PSEMU_ASSUMED_CPU_HZ` reference rate) instead of following `CLK_MODE`, the ratio between a given `CLK_MODE`'s real Hz and the reference rate predicts exactly the reported errors - mode 7 (~4MHz, active during HELLO/beep) is ~3.97x the reference rate, predicting HELLO would run ~3.97x too slow when Timer is decoupled (reported: "at least 4x too slow"); mode 4 (~524kHz, active on the date screen) is ~0.496x the reference rate, i.e. the reference rate is ~2.01x mode 4's real rate, predicting the blink would run ~2.01x too fast when Timer is decoupled (reported: "twice as fast"). Both matched almost exactly.

**Fixed for real this time:** Timer now follows raw, `CLK_MODE`-scaled `step_cycles` again (like the outer loop), confirmed correct: real timers are "clocked by the System Clock" (which genuinely varies with `CLK_MODE`) - not decoupled to a fixed reference rate. RTC and DAC remain decoupled (real 1Hz oscillator; fixed audio-output rate, respectively - unrelated to Timer's role). The Attempt-2 "beep too fast" complaint that originally motivated decoupling Timer is now believed to have actually been the separate DAC output-pacing bug that Attempt 2 fixed at the same time - not something Timer following `CLK_MODE` caused. See `test_timer_scales_with_clk_mode` and `test_clk_mode_keeps_rtc_dac_on_real_time` (split out from the old combined test) for regression coverage. Verified directly via `tools/date_screen_probe.c`: the blink interval roughly doubled (from ~3 frames to ~6-7 frames) after this fix, matching the predicted correction.

### IOP_STOP/IOP_START bit5 (Sound Enable)

While investigating the audio behavior above, found that real hardware requires audio to be enabled via **both** `DAC_CTRL` bit0 and `IOP_STOP`/`IOP_START` bit5 ("Sound Enable") at `0x0D800004`/`0x0D800008` - a register this emulator didn't model at all (writes silently discarded). Implemented (`core/src/iop.c`/`iop.h`): `IOP_STOP` ORs bits into a shared bitmask, `IOP_START` ANDs them out (matching the documented set/clear semantics), and `dac_set_iop_muted` mirrors bit5's state directly into the DAC so `dac_tick` requires both gates open to output non-silent samples. ### Investigated: "Chocobo World plays no sound"

Traced a user report of no audible sound from Chocobo World (the app in `testdata/extracted/flash_app1.bin`) with a temporary write trace on `Timer1`'s control register, `DAC_DATA`, and `IOP_STOP`/`IOP_START` (same pattern as `psemu_clk_trace_enabled`, since removed) through the full `button_sim=3`/200M-instruction launch sequence. Findings:

- **The DAC output pipeline itself works correctly** - not in question, since the BIOS's own boot beep (already tuned/validated earlier this session) proves it end-to-end through real hardware listening tests.
- **Chocobo World's own FLASH1 code does actively try to play a sound**, not just the BIOS: shortly after entering FLASH1 (within its first ~3300 instructions), it writes `DAC_CTRL`=1, `IOP_START`=0x20 (clearing the sound-stopped bit), then loops writing `DAC_DATA` with cleanly alternating values that decode to +32704/-32768 - a valid full-amplitude square wave - for about 32 cycles before disabling both again. That's genuinely correct DAC usage, confirmed by manually decoding the raw bytes against the `DACV` bitfield.
- That one episode is the **only** DAC activity from the app anywhere in the 200M-instruction trace (which covers boot, generic launch, and a very large amount of subsequent execution) - it's brief (roughly 17ms of real time at the `CLK_MODE` active then), and nothing else in the run ever touches `DAC_CTRL`/`DAC_DATA` again. Most likely this is a one-time launch chime, easy to miss by ear, and the game's other sound (if any) is gated behind actual gameplay states/menus inside Chocobo World's own UI that this session's generic "launch the app" button simulation was never designed to reach (it repeats a BIOS-menu-shaped Down/Fire/Right/Fire pattern indefinitely once the app is already running, which doesn't correspond to real play within the game itself). Confirming or ruling this out needs real interactive play focused on getting further into the actual game, not more automated tracing of the generic launch path.
- **Found and fixed a real, confirmed bug along the way, independent of the above:** `iop_write8` only applied an `IOP_STOP`/`IOP_START` write's effect once a full 32-bit store's top byte arrived (`shift==24`) - a deferred-commit pattern copied from other registers without checking whether it was actually needed here. Direct tracing showed real BIOS and app code write both registers via **single-byte stores** (e.g. the app's own `IOP_START=0x20` write above), which never reach `shift==24` and were silently discarded entirely - `IOP_STOP`/`IOP_START` were completely inert for any single-byte writer. This happened not to change the outcome in the trace above (the default "not stopped" state already left the gate open, so the no-op bug was invisible there), but it's still wrong: a real single-byte `IOP_STOP` write meant to genuinely mute audio would never take effect. Fixed by applying each byte's OR/AND-NOT directly to `data` immediately (correct, since OR/AND-NOT are per-bit and commute across bytes - the deferred-commit step was unnecessary even for 32-bit writers). See `test_iop_stop_start_take_effect_via_single_byte_writes`.

### Investigated: Chocobo World crashes on getting into an event

Reported directly from interactive play: the game crashes after reaching an "event" (the map/story trigger screens Chocobo World shows periodically). Confirmed real user-described repro steps: on start, press Action to skip to the time-setting screen, Down to reach the year field, Action to confirm, Right to reach Chocobo World in the app list, Action to launch it, then wait - once the game settles into a holding pattern with the LCD apparently static (an event dialog waiting for input), pressing Action to continue is what triggers the crash.

**Reproduced deterministically**, though via a different path than the exact manual steps above: `tools/inspect.c` gained a new `button_sim=6` mode (kept in the tool, not removed - useful for future bug-hunting) that runs the same confirmed launch sequence once, then switches to a long, deterministically-seeded pseudo-random Up/Down/Left/Right/Fire "exploration" pattern for the rest of the run (unlike `button_sim=3`'s looping BIOS-menu-shaped input, which never varies and can't reach real gameplay states). A 3-billion-instruction run hit `cpu.unimplemented` at **instruction #1,319,429,329**, with the framebuffer showing what looks exactly like an event dialog (a bordered box with text) at the moment of the crash - strongly consistent with the reported symptom, confirmed 100% reproducible (same instruction number across repeated runs with the same seed).

Tracing the mechanism (after first fixing a bug in `tools/inspect.c`'s own crash reporter, which re-read the "raw opcode" from the unmasked, potentially-odd `pc` instead of the actual halfword-aligned fetch address - a real but harmless-to-emulation diagnostic-only bug, now fixed):

- The CPU is executing a legitimate function (reached via a normal, correctly-computed Thumb `BL` call) that starts with a couple of ordinary data-processing instructions, then runs straight into a 16-bit value (`0xFF1F`) that decodes - per the real ARMv4T Thumb ISA, where every `0xF000`-`0xFFFF` value is architecturally part of the two-halfword long-branch-with-link (format 19) encoding - as the **second half of a `BL`**, using whatever is currently in `LR` as the "high offset" base.
- `LR` at that point is stale: left over from a **previous, unrelated, already-completed** `BL` several instructions earlier, since nothing in between wrote to it. Combining that stale value with this instruction's own encoded low-offset computes a wild target (confirmed by hand: `stale_LR + (offset11<<1)` matches the observed jump exactly), landing the CPU a short distance away in what continues to look like plausible code for a few more instructions before finally hitting a genuinely unrecognized pattern and tripping `cpu.unimplemented`.
- This mechanism only makes sense if the real, straight-line control flow was never supposed to reach that `0xFF1F` halfword at all - real compiled Thumb code doesn't emit an orphaned second-half of a `BL` as live, reachable code. So something upstream (still unidentified) must be branching incorrectly or falling through where real hardware would have taken a different path. Given this is ~1.3 billion instructions into an otherwise entirely correct run (matching real hardware behavior this whole session, including a great deal of real gameplay), a generic, frequently-hit CPU decode bug seems unlikely - more likely either a rare, narrow ARM/Thumb emulation gap only reachable through this specific control-flow shape, or a hardware feature Chocobo World's event system depends on that isn't modeled at all yet (it's one of the few PocketStation titles known to use the IR module - "press Left/Right in the Map screen to go to the IR menu" - and this emulator's `ir.c` is a minimal, mostly-unvalidated stub).
- **Not yet root-caused further** - this needs either a proper disassembly of the surrounding function with real symbol/data-boundary information (to know whether a branch was skipped or the target was simply wrong), or narrowing down the pseudo-random input sequence that reaches it to something a human could replay step-by-step. Left as an open, actively-reproducible bug rather than guessed at further.

**Concrete improvement made regardless of the open root cause:** the CPU tripping `cpu.unimplemented` was previously completely invisible to the desktop frontend - `tools/inspect.c` already checked and reported it, but `frontends/desktop/main.c` never did, so real players would see this exact failure mode as an unexplained silent hang/freeze with the CPU still nominally "running" (in reality endlessly executing whatever garbage bytes it landed on, with no defined behavior). Added a public `psemu_cpu_faulted()` API (`core/include/psemu/psemu.h`/`core/src/psemu.c`) and wired it into the desktop frontend: once tripped, the frontend stops stepping the CPU (freezing on the last good frame, matching what a user already sees) but now also prints a clear diagnostic to stderr instead of continuing to silently corrupt state. See `test_cpu_faulted_flag`.

### Verified section-by-section against real hardware behavior

This implementation was independently verified section-by-section against real hardware behavior (checked directly rather than via a summarized web view, the same lesson learned from the earlier `DAC_DATA` bit-layout mistake) section-by-section against the current implementation: memory map, official register names/bit-fields, interrupt controller, timers, RTC, DAC, IOP, and the `F_BANK_VAL`/`F_BANK_FLG` mechanics just implemented above.

**Confirmed correct, no changes needed:** the interrupt controller's bit-to-source mapping in `intc.h` (`INT_BTN_*`/`INT_TIMER*`/`INT_RTC`/`INT_IOP`/`INT_IRDA`, and the `INT_IRQ_MASK`/`INT_FIQ_MASK`/`INT_STATUS_MASK` masks) matches the official 14-source table exactly, including which sources are FIQ (bit6 COM, bit13 Timer2) vs IRQ. Timer clock-divisor bits, `DAC_DATA`'s 10-bit signed `DACV` field, `IOP_STOP`/`IOP_START` bit5 polarity, RTC reset values/`PRGSEL`/`CNTSEL` semantics, and `F_BANK_VAL`/`F_BANK_FLG`'s `table[p]=v` mechanic all matched the official register tables precisely. The linked memory-card-directory-format page also confirmed the block-chain-walk logic (`flash_load_app`'s allocation-state/next-block-link parsing) is correct.

**Found and fixed: `LCD_MODE` (`0x0D000000`) was entirely unmodeled.** No bus handler existed for this address at all (only `LCD_VRAM` at `0x0D000100` was wired up) - reads/writes silently fell through to the bus's default no-op, and `psemu_get_framebuffer` returned raw VRAM unconditionally. Real hardware has documented bits here: bit6 `DISON` (display on/off) and bit7 `ROT` (rotate the display 180 degrees, meant to be kept in sync with `INT_INPUT.11`'s docking flag so the screen reads right-side-up whether handheld or docked). Implemented in `core/src/lcd.c`/`lcd.h`: a new `lcd_mode_read8`/`lcd_mode_write8` pair at `PSEMU_LCD_MODE_BASE`, and a `presented` buffer (recomputed from raw `vram` on every VRAM or `LCD_MODE` write) that `psemu_get_framebuffer` now returns instead of raw VRAM - blank when `DISON` is clear, 180-degree-rotated (reversed scanline order, each scanline's 32 bits reversed left-right) when `ROT` is set. Default `mode` has `DISON` set (not `ROT`) rather than the real, undocumented POR value, specifically chosen to preserve this emulator's previously-validated "always render VROM directly" behavior for any app that never touches this register - a deliberate safe default, not a hardware fact. See `test_lcd_mode_dison_and_rotate`. Verified via the full `button_sim=3`/150M-instruction regression: FLASH1/dispatch instruction counts are unchanged from every prior baseline, and the real BIOS does write to `LCD_MODE` during the run (a few extra framebuffer-dirty events appear versus the pre-change trace, from `LCD_MODE` writes themselves rather than VRAM changes) while the final rendered output is unaffected - confirming the real BIOS leaves `DISON` set and `ROT` clear for this boot sequence, so nothing observably changes for every scenario already tested this session.

**Confirmed real gaps, left unimplemented (no observed impact yet):**

- `BATT_CTRL` (`0x0D800020`, low-voltage detection) falls through to the bus's unmapped default (reads 0, writes no-op) - no real behavior depends on battery-level sensing in an emulator with no actual battery.
- `FLASH_CTRL+0`'s "always read back bit0=1" behavior (see the busy-wait-bits fix above) doesn't cleanly correspond to the official `REGRemap` register's documented `GENREM`/`FLASHVIR` bit semantics - it's an empirically-confirmed-necessary fix (verified by getting real BIOS execution past a hard hang), not one derived from the register's documented meaning. Recorded here so a future reader doesn't assume it's spec-accurate.
- `F_BANK_VAL`'s official notes explicitly call out that mapping *multiple physical blocks to the same virtual slot* has unknown real hardware behavior ("maybe the data becomes ANDed together"). `flash_resolve_physical_bank` just returns the lowest-indexed matching physical block in that case - a reasonable but arbitrary choice among the documented unknowns, not a confirmed-correct model.
- The real BIOS-mirrored-at-address-0 + `GENREM` remap dance (BIOS ROM aliased to `0x00000000` until the kernel writes `F_CTRL` bit0 to replace it with RAM) isn't literally modeled - `psemu_reset` just starts the CPU's PC directly at `PSEMU_BIOS_BASE` (`0x04000000`), skipping the pre-remap phase entirely. This is a deliberate, already-validated simplification (150M+ instructions of real BIOS+app execution with correct behavior), not an oversight - flagged here since it's a real deviation from the literally-documented mechanism, kept because changing it would add complexity with no demonstrated behavioral benefit.
